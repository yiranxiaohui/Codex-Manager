# Codex 对齐会话语义与自动切换设计

## 目标

本文档定义 CodexManager 后续网关行为的目标语义：

- 以官方 Codex 的会话心智为准，不再以旧兼容行为作为主路径语义。
- 同一个本地会话在任一时刻只绑定一个活动账号。
- 手动切换账号会切线程，自动切换账号也必须切线程。
- 不再允许把同一条上游线程状态跨账号复用。
- 账号轮询只负责“首次命中”与“重新绑定”，不再参与同一会话的每轮请求分发。

本文档是完整设计文档，不是最小实现方案。

## 背景

当前实现已经大量偏向 `https://chatgpt.com/backend-api/codex`，但仍保留了一部分旧兼容路径思路：

- `conversation_id` 会被补成 `prompt_cache_key`，并参与 `session_id` 对齐。
- 账号选路仍以 `key_id + model` 为主维度，不理解“同一会话应固定账号”的语义。
- failover 时会剥离会话粘性，优先保可用性，而不是维持 Codex 风格的线程一致性。
- 历史兼容开关曾影响主线路径语义。

这套混合模型能提升兼容面，但副作用也很明显：

- 同一 `conversation_id` 可能被多个账号先后承接。
- Prompt cache 命中率不稳定。
- 会话线程与账号路由没有单一事实源。
- 程序在部分环境下仍会触发 Cloudflare，而官方 Codex 相对稳定。

因此后续方向应明确收敛为 `Codex-first`，而不是继续围绕旧兼容逃生阀打补丁。

## 问题定义

当前最核心的问题不是“能不能多账号共用同一个 `conversation_id`”，而是“系统是否允许同一个上游线程跨账号漂移”。

对齐 Codex 后，应该遵守以下事实：

1. 本地会话可以持续存在。
2. 但上游线程状态不能跨账号复用。
3. 一旦账号发生切换，不论是手动还是自动，都必须视为进入新的上游线程世代。
4. 同一会话后续请求，必须优先复用当前绑定账号，而不是重新参与全局轮询。

换句话说：

- 可以保留同一个本地 `conversation_id`
- 但不能继续沿用旧账号上的 `prompt_cache_key`、`session_id`、`turn-state`
- 切账号就切线程

## 非目标

本文档不追求以下目标：

- 不继续保证旧兼容模式作为长期主路径存在。
- 不支持“同一条上游线程跨多个账号无缝延续”。
- 不承诺仅靠应用层语义对齐就完全消除 Cloudflare；传输层指纹仍需单独评估。
- 不以保留所有历史环境变量和 UI 开关为优先目标。

## 当前实现摘要

### 已有能力

- `/v1/responses` 已经会改写到 `backend-api/codex` 兼容形态。
- `conversation_id` 已能注入到 `prompt_cache_key`。
- 上游 `session_id` 已能与 `conversation_id` / `prompt_cache_key` 对齐。
- 主链路已关闭 OpenAI fallback 主路径。

### 当前不符合目标的点

- 路由维度仍是 `key_id + model`，不是“会话绑定账号”。
- `conversation_id` 当前更像“线程锚点候选”，不是“会话绑定主键”。
- failover 后默认会把同一会话发往新账号，但缺少显式“线程换代”状态。
- 历史兼容路径已经开始移除，但会话绑定语义尚未完全替代旧轮询行为。
- 全局“手动优先账号”覆盖范围过大，不适合会话级绑定模型。

## 术语

### 本地会话 ID

由客户端提供或本地生成的 `conversation_id`。它代表产品层会话，不直接等同于上游线程。

### 会话绑定

指 `(platform_key, local_conversation_id)` 当前绑定到哪个 `account_id`。

### 线程世代

同一会话在不同账号、或同一账号上的重建线程，都视为不同世代。世代单调递增，用于明确“切账号就切线程”。

### 有效线程锚点

发往上游时真正使用的线程标识，记为 `effective_thread_anchor`。它不再直接等于原始 `conversation_id`，而是与当前绑定账号和线程世代绑定。

### 自动切换

不是用户手动点选，而是运行时因为账号失效、认证失败、模型不支持、挑战拦截、上游不可用等原因，主动将会话迁移到下一个账号。

## 目标语义

### 总体规则

1. 同一个本地会话在任一时刻只能绑定一个账号。
2. 绑定存在时，后续请求优先走绑定账号，不再重新参与 `ordered` / `balanced` 正常轮询。
3. 当账号切换发生时，必须创建新的线程世代和新的 `effective_thread_anchor`。
4. 旧账号上的 `session_id`、`prompt_cache_key`、`turn-state` 不得继续透传到新账号。
5. `ordered` / `balanced` 只用于：
   - 新会话首次绑定
   - 当前绑定失效后的重新绑定

### 关于“自动切换也要对应切换”

这是本设计的硬约束。

自动切换一旦发生，系统必须同步执行以下动作：

- 更新会话绑定的 `account_id`
- 线程世代 `thread_epoch + 1`
- 生成新的 `effective_thread_anchor`
- 丢弃旧 `turn-state`
- 停止复用旧账号相关的上游线程头

这意味着：

- 自动切换不是“同一线程换个账号继续跑”
- 自动切换是“同一本地会话切到新账号，并开始新的上游线程世代”

## 状态模型

### 绑定主键

会话绑定主键定义为：

```text
(platform_key_hash, local_conversation_id)
```

说明：

- 绑定不再以 `model` 为主键。
- 同一会话即使中途改模型，也优先尝试当前绑定账号。
- 若当前绑定账号不支持目标模型，再触发重新绑定。

### 绑定记录

建议新增持久化表 `conversation_bindings`，核心字段如下：

```text
platform_key_hash        TEXT    NOT NULL
conversation_id          TEXT    NOT NULL
account_id               TEXT    NOT NULL
thread_epoch             INTEGER NOT NULL
thread_anchor            TEXT    NOT NULL
status                   TEXT    NOT NULL
last_model               TEXT    NULL
last_switch_reason       TEXT    NULL
created_at               INTEGER NOT NULL
updated_at               INTEGER NOT NULL
last_used_at             INTEGER NOT NULL
PRIMARY KEY (platform_key_hash, conversation_id)
```

### 持久化选择

绑定需要落库，而不是只放内存，原因如下：

- service 重启后仍应延续当前会话对同一账号的偏好。
- 排障时需要可追溯“这条会话为什么切到了另一个账号”。
- 绑定语义已经是产品层事实，不再只是瞬时运行时状态。

### 清理策略

- 长时间未使用的绑定可按 `last_used_at` 定期清理。
- 默认建议保留 7 天。
- 账号被删除、禁用或 token 永久失效时，应同步删除或失效相关绑定。

## 路由与绑定规则

### 首次绑定

当 `(platform_key_hash, conversation_id)` 尚无绑定时：

1. 先按当前候选池过滤可用账号。
2. 再应用 `ordered` 或 `balanced` 选择首个账号。
3. 选中的账号写入 `conversation_bindings`。
4. 初始化 `thread_epoch = 1`。
5. 生成新的 `thread_anchor`。

### 绑定命中

当会话已存在绑定时：

1. 先检查绑定账号是否仍然可用。
2. 若可用，则直接命中，不再参与普通轮询。
3. 更新 `last_used_at`、`last_model`。

### 自动切换触发条件

以下情况允许自动切换，并要求同步切线程：

- 账号状态不是 `active`
- token 缺失或鉴权失败
- 上游返回 `401` / `403`
- 上游返回 challenge 类错误
- 上游返回 `429`
- 上游返回不可恢复 `5xx`
- 账号不支持当前请求模型
- 账号被 cooldown 判定为不可继续承担该会话

### 不应触发自动切换的场景

以下场景不应直接把同一会话切去别的账号：

- 同一会话已有正在执行中的请求
- 仅因为全局并发上限暂时已满
- 同一会话的下一轮请求只是比上一轮更早到达

原因：

- 同一会话并发扇出到多个账号，会直接破坏 Codex 风格的线程连续性。
- 这里应优先做“会话级串行化”或短暂等待，而不是切账号。

### 手动切换

当用户显式切换当前会话账号时：

1. 新账号必须通过候选可用性校验。
2. 更新绑定到新账号。
3. `thread_epoch + 1`
4. 生成新的 `thread_anchor`
5. 旧 `turn-state` 作废

## 上游线程规则

### 新规则

后续发往上游时，应以绑定态为单一事实源：

- `/v1/responses`：`prompt_cache_key = effective_thread_anchor`
- `session_id = effective_thread_anchor`
- `x-codex-turn-state` 只允许在当前绑定账号、当前线程世代内继续使用

### 明确禁止

以下行为必须禁止：

- 直接把原始 `conversation_id` 作为跨账号共享线程锚点
- 自动切换后继续透传旧 `session_id`
- 自动切换后继续透传旧 `turn-state`
- 自动切换后继续沿用旧账号产生的 `prompt_cache_key`

### `conversation_id` 的角色变化

在目标设计下，`conversation_id` 的语义是：

- 本地产品会话主键
- 绑定查找键

它不再是：

- 上游跨账号共享线程锚点
- 旧会话头的直接透传值

## Prompt Cache 语义

### 对齐原则

Prompt cache 的复用前提应是：

- 同一账号
- 同一线程世代
- 同一前缀

因此 `prompt_cache_key` 不应再简单等于原始 `conversation_id`，而应等于绑定态生成的 `effective_thread_anchor`。

### 这样做的结果

- 同账号连续多轮对话更稳定命中 cache
- 账号自动切换后不再错误复用旧 cache
- 会话语义与账号语义保持一致

### 取舍

这样会主动放弃“跨账号复用同一个 cache key”的幻想，但这正是目标行为，因为该行为与 Codex 风格线程语义冲突。

## 配置与 UI 调整

### 应删除或降级的主线概念

- 旧兼容开关
- 对应设置页入口
- 文档中以旧兼容路径作为主要设计依据的表述

补充约束：

- 迁移期间的文档、UI 文案和排障建议都不应再推荐这些开关。

### 仍然保留的概念

- `ordered` / `balanced`

但其含义需要收窄为：

- 新会话首次绑定策略
- 已绑定会话重新绑定时的候选排序策略

### 需要重定义的概念

现有“手动优先账号”是全局概念，和会话级绑定冲突。后续应改成两层语义之一：

1. 新会话默认优先账号
2. 当前会话显式切换账号

不应继续保留“一个全局开关覆盖所有会话”的语义。

## 传输层对齐说明

本设计文档主要约束应用层会话、路由和 header 语义。

但必须明确：

- 即使应用层行为完全对齐 Codex
- 只要上游 transport 指纹与官方实现差异较大
- 仍然可能在部分环境下被 Cloudflare 区分

因此后续工作需要拆成两条线并行推进：

### 语义线

- 会话绑定
- 自动切换即切线程
- 去旧兼容主线化

### 传输线

- 评估当前 `reqwest + rustls` 的 TLS / HTTP 指纹差异
- 评估 Cookie、连接复用、重试行为与官方客户端的偏差
- 必要时单独替换或收敛上游 transport 实现

本次设计先把语义线定死，不把传输层问题与会话语义继续混在一起。

## 验收标准

### 行为验收

1. 同一 `conversation_id` 在账号健康时连续多轮请求，必须命中同一账号。
2. 手动切换账号后，下一轮请求必须使用新账号，且使用新的线程锚点。
3. 自动切换账号后，下一轮请求必须使用新账号，且使用新的线程锚点。
4. 自动切换后不得继续透传旧账号上的 `session_id` 与 `turn-state`。
5. 同一会话并发请求不得扇出到多个账号。
6. `ordered` / `balanced` 只影响首次绑定和重新绑定，不影响已绑定会话的稳定命中。

### 测试验收

至少需要补齐以下测试：

- 会话首次绑定测试
- 会话绑定命中测试
- 自动切换后线程世代递增测试
- 手动切换后线程世代递增测试
- 旧 `turn-state` 不跨账号透传测试
- 同会话并发不跨账号测试
- `compact` 路径与标准 `responses` 路径一致性测试
- service 重启后绑定续用测试

### 文档验收

需要同步更新：

- `docs/report/*FAQ*`
- `docs/report/*环境变量*`
- 设置页帮助文案
- `CHANGELOG.md`

## 迁移与发布要求

### 数据迁移

- 增加 `conversation_bindings` 表
- 增加必要索引
- 升级后不尝试迁移旧内存态会话
- 首次落到新版本时，旧会话会按新规则重新建立绑定

### 发布要求

- 发布说明必须明确：会话与账号关系已从“请求级轮询”切换为“会话级绑定”
- 明确说明：自动切换账号会重建上游线程
- 明确说明：旧兼容开关已删除，不再作为设置项或环境变量暴露

### 回滚要求

若上线后发现重大兼容问题，回滚方式应为版本回滚，而不是重新恢复旧兼容路径作为长期主路径语义。

## 决策结论

最终决策如下：

1. CodexManager 后续以 `Codex-first` 为唯一主路径语义。
2. 同一会话只绑定一个活动账号。
3. 手动切换账号会切线程，自动切换账号也会切线程。
4. 原始 `conversation_id` 不再直接充当跨账号共享的上游线程锚点。
5. `ordered` / `balanced` 退化为首次绑定与重新绑定策略。
6. 旧兼容开关不再作为长期主线路径保留。

这份文档是后续代码改造、测试补齐、文档更新与发版说明的共同事实源。
