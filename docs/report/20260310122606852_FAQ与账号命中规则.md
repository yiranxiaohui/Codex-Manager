# FAQ 与账号命中规则

## 常见问题
- 授权回调失败：优先检查 `CODEXMANAGER_LOGIN_ADDR` 是否被占用，或在 UI 使用手动回调解析。
- 模型列表或请求被挑战拦截：优先检查代理出口、请求头差异和 `CODEXMANAGER_UPSTREAM_FALLBACK_BASE_URL`，不再通过上游 Cookie 排障。
- 仍被 Cloudflare/WAF 拦截：不要再沿旧兼容路径排障，后续只按 `Codex-First` 语义推进。
- “部分数据刷新失败，已展示可用数据”频繁出现：自动刷新场景已改为仅记录日志；手动刷新会提示失败项与示例错误。优先检查设置页“后台任务”间隔/开关，以及 service 日志中的失败任务名。
- 独立运行 service/Web：若所在目录不可写（如安装目录），请设置 `CODEXMANAGER_DB_PATH` 到可写路径。
- macOS 代理环境下请求 `502/503`：优先确认系统代理未接管本地回环请求（`localhost/127.0.0.1` 走 `DIRECT`），并确保地址使用小写 `localhost:<port>`。

## 迁移说明

### Codex-First 方向

当前仓库已经确定后续方向为 `Codex-First`：

- 不再以旧兼容行为作为主线路径
- 同一会话只绑定一个活动账号
- 手动切换账号会切线程，自动切换账号也会切线程

对应设计文档：

- [20260319195028315_Codex对齐会话语义与自动切换设计.md](../plan/20260319195028315_Codex对齐会话语义与自动切换设计.md)

### 当前行为与目标行为的区别

当前行为：

- `balanced` 仍按 `Key + 模型` 严格轮询
- 会话与账号的绑定语义仍在从“按请求轮询”迁移到“按会话绑定”中

目标行为：

- 已绑定会话优先命中绑定账号，不再参与每轮普通轮询
- 自动切换账号会同步切换上游线程世代
- 旧兼容开关完全退出主路径，且不再作为推荐配置使用

## 账号命中规则
- `ordered`（顺序优先）模式下，网关按账号 `sort` 升序构建候选并依次尝试，例如 `0 -> 1 -> 2 -> 3`。
- 这表示“按顺序尝试”，不是“永远命中 0 号”；前序账号若不可用或失败，会自动切到下一个。

### 前序账号不被命中的常见原因
- 账号状态不是 `active`
- 账号缺少 token
- 用量判定不可用，例如主窗口已用尽、用量字段缺失
- 账号处于 cooldown 或并发软上限触发跳过

### `balanced` 模式
- `balanced`（均衡轮询）模式默认会按 `Key + 模型` 维度在所有可用账号间严格轮转，不保证从最小 `sort` 开始。
- 仅在你显式调大 `CODEXMANAGER_ROUTE_HEALTH_P2C_BALANCED_WINDOW` 时，才会在均衡轮询头部再叠加健康度换头。

## 排障日志
可查看数据库同目录 `gateway-trace.log`：
- `CANDIDATE_POOL`：本次请求候选顺序
- `CANDIDATE_START` / `CANDIDATE_SKIP`：实际尝试与跳过原因
- `REQUEST_FINAL`：最终命中账号
